# ADR-0001：模块调用采用内部 Tool Registry，并预留 MCP Adapter

- 状态：已接受
- 日期：2026-06-29

## 背景

XuanAI 将逐步加入 Todo 之外的多个模块，并支持桌面端、UniApp 等客户端。需要让不同 AI 厂商以统一方式调用模块，同时避免业务模块依赖特定模型协议。

## 决策

模块能力先注册到客户端内的 Tool Registry，并转换为 OpenAI 兼容 function tools。跨进程、跨设备或远程部署出现实际需求时，在相同工具契约外增加 MCP Adapter。Skill 只用于编排工具和沉淀工作流。

Tool Registry 统一负责：

- 工具发现和名称唯一性；
- JSON Schema 暴露；
- 风险级别与权限确认；
- 参数校验和统一结果；
- 后续审计与 MCP 映射。

## 原因

- 当前 Todo 数据位于客户端本地，引入独立 MCP Server 会增加部署和通信成本。
- 内部调用可以先验证产品体验，同时不牺牲协议边界。
- OpenAI 兼容工具定义能复用多数厂商现有接口。
- MCP Adapter 可以复用同一套工具名称、Schema 和结果格式。

## 后果

- 桌面端和 UniApp 分别实现轻量注册表，但共享同一份工具契约文档。
- 所有写入操作都经过权限门，AI 不能直接修改本地存储。
- 兼容端点若不支持 function tools，将明确提示能力不受支持。
- 后续新增模块必须先定义工具契约，再接入 AI。

## 暂不采用

- **所有模块立即做成 MCP Server**：当前本地模块收益不足以覆盖部署复杂度。
- **只使用 Skill 调模块**：Skill 更适合编排，无法替代稳定的可执行能力边界。
- **让 AI 直接操作 Store**：难以授权、审计、测试和跨端复用。
