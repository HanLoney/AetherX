# AetherX 家庭节点与多端同步架构

## 目标

AetherX 不建设官方数据云。每位用户拥有自己的 AetherX Hub，Hub 可以随桌面安装包运行，也可以由用户部署到 NAS、树莓派或私人服务器。桌面端和手机端只通过 Hub 的 API 访问数据。

第一阶段优先解决以下问题：

- 普通用户安装桌面程序后，不需要单独安装 Node.js 或手动启动后端；
- 手机与电脑处于同一局域网时可以安全配对；
- 电脑端产生的变化可以实时通知手机；
- 手机断线后可以从上次游标继续补拉变化；
- 所有设备继续沿用现有的账号隔离，不能跨账号读取数据；
- SQLite 只由 Hub 所在设备写入，不同步正在使用的数据库文件。

## 核心决策

```text
┌─────────────────────── 电脑 ───────────────────────┐
│ Electron 桌面界面                                     │
│          │ REST                                       │
│          ▼                                            │
│ AetherX Hub（托盘常驻、可开机启动）                     │
│   ├─ 身份认证与设备授权                                 │
│   ├─ Agent Loop、工具与授权状态机                        │
│   ├─ 业务 API                                          │
│   ├─ 增量变更日志                                      │
│   ├─ SSE 实时通知                                      │
│   └─ SQLite（唯一写入源）                              │
└────────────────────────┬─────────────────────────────┘
                         │ HTTPS / 私人网络
                         ▼
                  Capacitor 手机客户端
```

### 单写原则

Hub 是数据的唯一权威来源。桌面端和手机端不直接打开数据库，也不通过 Syncthing、网盘或共享目录同步 `xuanai.db`。所有写入都经过业务 API。

聊天也遵循相同原则：两端只调用 Agent API，提示词、记忆召回、工具循环和会话保存
均由 Hub 统一完成。同一会话同时只处理一个活跃运行，避免多设备写入乱序。

第一版手机端以在线客户端为主，可以保留只读缓存；离线写入、冲突合并和多主复制不属于第一阶段。

### 实时同步原则

实时同步由两部分组成：

1. REST API 负责读取和修改数据；
2. SSE 通道只发送变化通知，客户端收到通知后按游标拉取增量变化。

变化通知不携带聊天正文、记忆内容、图片或密钥。这样可以避免敏感数据进入长连接日志，也能保证断线恢复与实时通知使用同一套增量语义。

### 部署模式

| 模式 | 数据位置 | 适用场景 | 限制 |
| --- | --- | --- | --- |
| 桌面家庭节点 | 用户电脑 | 默认安装方式 | 电脑关闭后手机无法同步 |
| 局域网常驻节点 | NAS、树莓派 | 家庭内持续在线 | 用户需要自备设备 |
| 私人远程节点 | 自有服务器 | 随时随地访问 | 用户需要维护服务器与 HTTPS |
| 私人组网 | WireGuard、Headscale | 不公开暴露 Hub | 需要用户自建控制节点或 VPN |

AetherX 不承诺在“没有公共服务、任意 NAT、完全零配置”三个条件同时存在时实现外网直连。默认产品体验先覆盖局域网，远程访问作为可选自托管能力。

## 桌面端运行模型

桌面安装包携带后端源码，并使用 Electron 自带的 Node.js 运行时启动 Hub。开发环境仍可连接已经启动的 `127.0.0.1:4318`，避免重复占用端口和迁移现有开发数据。

运行规则：

- 本地地址且端口没有可用的 AetherX Hub 时，桌面端启动内置 Hub；
- 本地地址已有健康的 AetherX Hub 时，复用现有进程；
- 配置为远程服务器地址时，不启动内置 Hub；
- 关闭主窗口时隐藏到托盘；
- 只有托盘中的“退出”操作才真正停止桌面程序及其内置 Hub；
- 开机启动由用户显式开启。

内置 Hub 的新数据默认存放在 Electron `userData/hub` 下。已经通过 `AETHERX_DATA_DIR` 指定数据目录的安装继续使用该目录，不静默搬动数据库和加密材料。

## 设备配对协议

配对采用短时、一次性会话，不复用账号密码。

1. 已登录的桌面端创建配对会话；
2. Hub 返回随机会话 ID、256 位随机秘密和过期时间；
3. 桌面端将连接地址、会话 ID 和秘密编码进二维码；
4. 手机提交设备名称和设备公钥，状态变为 `pending`；
5. 桌面端展示设备请求，用户确认后状态变为 `approved`；
6. 手机凭一次性秘密兑换设备令牌；
7. Hub 只保存令牌哈希，明文令牌只返回一次；
8. 用户可以随时撤销设备，撤销后令牌立即失效。

二维码和兑换请求必须通过受信任的 HTTPS、证书固定连接或私人 VPN 传输。发布版本不把局域网明文 HTTP 作为默认安全方案。

## 数据模型

```text
paired_devices
- id
- user_id
- name
- public_key
- token_hash
- status
- created_at
- last_seen_at
- revoked_at

device_pairing_sessions
- id
- user_id
- secret_hash
- status
- device_name
- public_key
- expires_at
- created_at
- approved_at
- redeemed_at
- device_id

sync_changes
- seq
- user_id
- entity_type
- entity_id
- operation
- created_at
```

`seq` 是单调递增的全局游标。客户端只保存自己最后确认的 `seq`，重新连接后请求 `after=<seq>` 即可补齐遗漏。

## API 契约

```text
POST   /api/v1/pairing/sessions
POST   /api/v1/pairing/sessions/:id/claim
GET    /api/v1/pairing/sessions/:id
POST   /api/v1/pairing/sessions/:id/approve
POST   /api/v1/pairing/sessions/:id/redeem

GET    /api/v1/devices
DELETE /api/v1/devices/:id

GET    /api/v1/sync/changes?after=<seq>&limit=<n>
GET    /api/v1/sync/events?after=<seq>

POST   /api/v1/agent/chat
POST   /api/v1/agent/runs/:id/approve
```

设备令牌与普通会话令牌都使用 Bearer 认证。服务端从令牌解析 `user_id`，禁止客户端在业务请求中自行声明用户身份。

## 增量日志

第一阶段使用 SQLite 触发器记录用户业务表的新增、更新与删除。触发器与业务写入处于同一事务，因此不存在“数据已提交但变化记录丢失”的窗口。

同步日志只表达“哪个实体发生了什么变化”，不保存完整数据快照。图片继续按普通资源请求读取，后续再迁移到内容寻址文件和缩略图接口。

## 安全边界

- AI API Key 只保存在 Hub，不下发给手机；
- 设备令牌独立、可撤销，数据库只保存哈希；
- 配对秘密具有短过期时间且只能兑换一次；
- SSE 与增量查询都必须鉴权；
- 所有查询按认证上下文中的 `user_id` 过滤；
- 局域网监听默认关闭，只有用户开启“连接手机”后才绑定局域网接口；
- 开启局域网时必须展示系统防火墙和本地网络权限说明；
- 备份使用 SQLite Backup API、`VACUUM INTO` 或停机快照，不复制活跃数据库文件。

## 分阶段交付

### 第一阶段：电脑是家庭节点

- 桌面安装包携带并启动 Hub；
- 托盘常驻和开机启动；
- 设备配对、确认、兑换与撤销；
- 增量日志、断线补拉和 SSE 通知；
- Android Capacitor 客户端（已实现登录、核心页面、Keystore 凭证保护、增量补拉与 SSE 重连）；
- 局域网发现与二维码兜底。

### 第二阶段：可靠离线与远程访问

- 手机只读缓存和离线写入队列；
- 幂等请求 ID；
- 实体版本与冲突提示；
- WireGuard、Headscale 和自有域名接入指引；
- Docker Compose、NAS 和树莓派部署包。

### 第三阶段：多平台完善

- iOS 客户端；
- 内容寻址图片存储和缩略图；
- 自动备份、导入导出与恢复演练；
- 多节点复制的可行性评估。

## 不采用的方案

- **直接同步 SQLite 文件**：无法可靠处理 WAL、文件锁、并发写入和部分复制；
- **第一版使用 CRDT**：当前业务模型和单机 SQLite 架构不需要承担多主合并复杂度；
- **手机保存 AI 密钥**：扩大泄露面并造成多端配置漂移；
- **把设备令牌放进二维码**：二维码泄露会直接获得长期访问权限；
- **默认向公网开放 4318 端口**：缺少 TLS 和访问控制时风险不可接受。
